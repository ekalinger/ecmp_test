# ECMP TEST
***Схема:***

![Схема](https://github.com/ekalinger/ecmp_test/tree/fix-test/fix-test/network.png)
  
R1, R2 - namespaces  
input, OSPF - линки veth(3 штуки)  
пакеты забираются внутри namespace r2 с непосредственно veth  
  
Проверка работы ECMP основана на генерации пакетов с случайными последовательностями TCP портов, IP Source. IP Destination оставляем статичным, но при версии ядра 6.14 и выше можно протестить несколько IP destination  
Для проведения тестов генерируем пакеты и тут же вычисляем хэш и смотрим как он должен распределиться. Далее отправляем пакеты и снифаем уже на veth R2 (output1, output2, output3)  
Если распределения примерно совпадают, то тест проходит  
  
***Условия:*** 
Linux debian или ubuntu  
На текущий момент в ядре линукс старше 5.13 реализован хеширование пакета по чистому IP Source в случае активации ECMP, все версии ниже можно только настроить хеширование по IP Source и IP Destination  
Перел запуском настройки окружени необходимо проверить версию ядра и в зависимости от этого отредкатировать файл environment.sh и закомментировать/расскомментировать сответсвующие строки  
  
***Настройка окружения:***  
1)отредактировать при необходимости файл environment.sh и запустить его  
2)запустить pip3 install requirements.txt  
3)запустить тест pytest --alluredir=allure-results  
4)проверить получение маршрутов OSPF и установление метрик в namespaces r1 (ip netns exec r1 ip route)

10.0.1.0/24 dev veth-r1-input proto kernel scope link src 10.0.1.1  
10.1.1.0/24 dev veth-r1-l2 proto kernel scope link src 10.1.1.1  
10.2.1.0/24 dev veth-r1-l3 proto kernel scope link src 10.2.1.1  
10.3.1.0/24 dev veth-r1-l4 proto kernel scope link src 10.3.1.1  
172.172.174.1 proto ospf metric 20  
        nexthop via 10.1.1.2 dev veth-r1-l2 weight 1  
        nexthop via 10.2.1.2 dev veth-r1-l3 weight 1  
        nexthop via 10.3.1.2 dev veth-r1-l4 weight 1  
"""  
5)для проверки результата запустить allure serve -h xx.xx.xx.xx -p 10050 allure-results, где хх.хх.хх.хх IP адресс вашего сервера  

***Методики тестирования***
Проверка распределения пакетов по Source IP на 3 интерфейса.
Начальные условия - маршруты в default namespace, r1 и r2 должны быть установлены. Сетевая связность между сетями  10.10.10.0/24 и 172.172.174.0/24 должна быть обеспечена

ECMP необходимо проверить как с OSPF так и с BGP по всем пунктам ниже

1)На вход (на интерфейсе veth в default namespace) подаём траффик c разными IP Source адресами, входящих в подсеть 10.10.10.0/24, в namespace r2 на входных интерфейсах veth необходимо записать траффик в разные файлы, соответсвующего линкам

Критерии приёмки:
        а)в сумме по разным интерфейсам количество принятых пакетов должно быть равным отправленным
        b)внутри каждого файла с выходного интерфейса должно быть примерно равное количество пакетов, равное количеству пакетов отправленных/количество активных линков между r1 и r2 (расхождение примерно 10-15%)
        c)в каждом выходном файле не должно быть дублированных пакетов как внутри самого дампа, так и между этими файлами
        d)В скрипте TCP порты для каждого пакета устанавливаются рандомно(т.е. на каждый запуск будет сформирован случайные порты), но при нескольких прогонах, пакеты внутри всех файлов каждого прогона должны быть одинаковы 

!!! файлы с предыдущего задания необходимо сохранить

2)Выключаем один линк на выбор, на вход подаём траффик c разными IP Source адресами, входящих в подсеть 10.10.10.0/24, в namespace r2 на входных интерфейсах veth необходимо записать траффик в разные файлы, соответсвующего линкам.

Критерии приёмки:
        а)в сумме по разным интерфейсам количество принятых пакетов должно быть равным отправленным
        b)внутри каждого файла с выходного интерфейса должно быть примерно равное количество пакетов, равное количеству пакетов отправленных/количество активных линков между r1 и r2 (расхождение примерно 10-15%)
        с)в каждом выходном файле не должно быть дублированных пакетов как внутри самого дампа, так и между этими файлами
        d)в каждом файле должны быть пакеты соответсвено из фала выше теста + распределённые из отключенного соединения

3)Включаем линк, проверяем восстановление путей, в namespaces r2 включем сервис nc -l 172.172.174.1 1234. В default namespaces вводим команду "nc 172.172.174.1 1234" и отправляем любой текст, в консоли на r2 должны получить этот текст. Определяем по какому линку (можно по наличию этого IP в файлах выше) прошло соединение nc, выключаем любой другой линк, пытаемся отправить сообщение в эту же сессию

Критерии приёмки:
        a)после выключения линка соединение nc сервер и клиент не должно быть разорвано, сообщения должны отправляться

4)Включаем линк, проверяем восстановление путей, подаём траффик c разными IP Source адресами, входящих в подсеть 10.10.10.0/24, в namespace r2 на входных интерфейсах veth необходимо записать траффик в разные файлы, соответсвующего линкам. Добавляем ещё один линк, по количеству линков он должен превышать максимально разрешённых. Подаем ещё раз аналогичный траффик.

Критерии приёмки:
        а)в сумме по разным интерфейсам количество принятых пакетов должно быть равным отправленным
        b)проверить в ip route количество маршрутов - их должно быть 3.
        c)траффик должен проходить только по 3 линкам.
